# 融合指标说明:我们到底在测什么

## 0. 先讲清楚要回答的问题

一个 VLM 里,图像 token 和文本 token 在 transformer 主干里一层层往上走。
**"vision 和 text 在哪一层开始交互"** 听起来是一个问题,其实是三个不同的问题,
答案层数不一样:

| 问法 | 直觉 | 我们的指标 |
|---|---|---|
| ① 文本表征里**有没有**图像信息? | 哪怕没用,信息进来了没 | `rho(l)` |
| ② 把这层的跨模态连接**切掉**,答案会不会变? | 这层的交互**有没有用** | `cost(d)` / `nu(l)` |
| ③ 注意力**看没看**图? | 描述性,容易误导 | 注意力质量(旧探针) |

最重要的是 **②(功能必要性)**,因为它直接对应"如果把 vision 和 text 分开训练/
推理,会不会掉点"——这才是"early fusion 有没有用"的真问题。①是"信息在不在",
②是"信息有没有被用上",两者可以差很远(信息早早进来了,但要到很深的层才真正
被用来出答案)。

---

## 1. 一切的基础:两种"手术"

所有指标都建立在两个操作上。模型输入 = `[图像 patch] + [问题文字] + [答案位]`。

- **屏蔽(block)**:在某些层里,把"文本 token 去看图像 token"这条注意力边**切断**
  (mask 成 −∞)。文本在这些层里就"瞎了",看不到图。
- **换图(swap)**:把图像换成**另一道题的图**(内容错的图),问题不变。
  这是"零假设"——图像在,但内容对这道题没用。

有了这两把刀,就能问:切了/换了之后,模型答得对不对、概率分布变了多少。

---

## 2. 指标一:`cost(d)` 和 `retained(d)` —— 功能融合在哪(最重要)

### 怎么测
从第 0 层到第 d 层,**全部**切断文本→图像的注意力(累积屏蔽 `[0..d)`),
看 VMCBench 准确率。d 从 1 扫到 28(每层都扫)。

- `cost(d)` = 屏蔽前 d 层后的准确率(用**真图**)。
- `cost_null(d)` = 同样屏蔽,但用**换的错图**(零假设)。
- **`retained(d) = cost(d) − cost_null(d)`** = 屏蔽掉前 d 层的跨模态读取后,
  **还活着的"可用图像信号"**。

### 怎么读
- `retained(0)` = `intact − swap` = `R0`,模型全部的可用图像信号
  (base = +2.4pt:用对图比用错图只高 2.4 个点——这个模型本来就很依赖文本先验)。
- 随着 d 变大,如果某层的跨模态读取**对答案有用**,切掉它 `retained` 就下降。
- `retained` 从 R0 掉到 0 的那段层,就是**功能融合真正发生的地方**。

> bee-mix base:`retained` 从 d=1 的 +0.030 一路保持到 d≈7,然后 d8–d12
> 掉到 0。**结论:前 7 层切了无所谓,功能融合集中在 L8–L16。**

### 派生标量
- **`L_f50`(半信号深度)** = `retained` 首次跌破 `R0/2` 的深度
  (base=11:屏蔽前 11 层,抹掉了一半的可用图像信号)。
- **功能 `CoM`(融合质心)** = 用 `retained` 的**逐层下降量**(每多切一层损失多少
  信号)做加权平均层数。这是最稳健的"融合中心在第几层"的单一数字。
  **base = 15.7,aux = 15.1。**

## 3. 指标二:`rho(l)` —— 表征融合率(信息在不在)

### 怎么测
在**每一层**的输出,取答案位的隐状态向量 `h`,跑三遍:
- `h(I)` = 用真图
- `h(I')` = 用换的错图
- `h(∅)` = 完全没图(纯文本)

然后:

```
rho(l) = ‖h(I) − h(I')‖ / ‖h(I) − h(∅)‖
        = 「换内容引起的变化」/「有没有图引起的总变化」
```

### 怎么读
- 分母 = 这层表征里"图像带来的全部影响"(包括跟内容无关的,比如序列变长、
  位置偏移、格式)。
- 分子 = 只跟"**是哪张图**"有关的那部分变化 = 真正的**内容融合**。
- `rho ≈ 1`:表征强烈在乎是哪张图 → 真融合。
- `rho ≈ 0`:表征只在乎"有没有图",不在乎内容 → 不是内容融合(可能只是位置/长度效应)。

> bee-mix base:`rho` 在 **L1 就高达 0.846**,然后单调衰减到 L12 的 0.245,
> 之后是 ~0.25 的平台。
> **意思:图像内容从第一层就进了文本表征(①的答案=很早),但结合②可知这些早期
> 内容信息在功能上是冗余的(切了前 10 层不掉点)。** 衰减是因为越往上,跟内容无关
> 的图像处理(格式化答案槽)越积越多,内容那部分占比就被稀释到 ~25%。

### 派生标量
- **`rho@L1`** = 第一层的内容融合率(base=0.846)。
- **`rho floor`** = 衰减后的平台值(base 在 L28 ≈ 0.23)。

## 4. 指标三:`nu(l)` —— 单层因果必要性

### 怎么测
只切**第 l 层一层**的跨模态边(不是累积),测输出分布变了多少:

```
nu(l) = KL( 完整模型的答案分布 ‖ 只切第 l 层后的答案分布 )
```

KL 越大 = 这一层单独切掉对答案影响越大。

### 怎么读 / 为什么它几乎是 0
bee-mix 全 28 层的 `nu` 都 ≈ 0.002 ≈ 零假设。**这不是 bug,是结论:**
跨模态必要性是**冗余/分布式**的——切掉任何**单独**一层,下游的层会重新去读图
补上,所以单层切几乎无损。这正是为什么 `cost(d)` 要用**累积**屏蔽才看得到信号
(②的标准做法),也说明 **"逼某一层做融合"这个思路天然行不通**:没有任何单层是
不可替代的瓶颈。

## 5. 怎么用这套指标判断 aux 训练有没有制造 early fusion

如果 aux loss 真把融合**前移**了,应该看到:
1. 功能 `CoM` 变小(融合质心往浅层走);
2. `cost(d)` 曲线在更浅的 d 就开始掉(`L_f` 左移);
3. `rho@L1` 变大(浅层内容融合变强)。

实际(base → aux):

| | base | aux | 期望(若真前移) | 实测 |
|---|---|---|---|---|
| 功能 CoM | 15.7 | 15.1 | ↓明显 | **没动** |
| rho@L1 | 0.846 | 0.656 | ↑ | **反而↓** |
| R0 图像依赖 | +0.024 | +0.013 | — | **↓(更不看图)** |

三项全部指向同一结论:**aux 没有制造 early fusion,反而略微削弱了早层内容敏感度和
整体视觉依赖**——这就是它掉 2 个点的机制层面解释。

> ⚠️ 一个易误读点:`L_f50` base=11、aux=10,看着像"aux 更早"。其实是假象:
> aux 的总信号 R0 只有 base 一半、噪声大,`retained` 在 d10 偶然触 0 又反弹到
> d12 的 +0.022(见逐层表)。所以以**稳健的 CoM** 为准,不要读 L_f 的 1 层差异。

## 6. 一句话总览

- `rho(l)`:**信息在不在**(表征轴)。bee-mix:从 L1 就在,但是冗余的。
- `cost(d)`/`retained(d)`:**信息有没有被用上**(功能轴,最重要)。bee-mix:
  真正起作用在 L8–16,质心 ~L15.7。
- `nu(l)`:**有没有单层瓶颈**。bee-mix:没有,必要性是分布式的。
- 三者一致:这个模型的功能融合居于主干中段(相对深度 0.36–0.57,与 NEO/Gemma
  跨架构同位),aux 训练没能撼动它。

图:`neo_report/figures/fig_fusion_unified_beemix.png`(三联图,红=aux 蓝=base,
虚线=零假设带)。

## 7. Fusion Depth Index (FDI):把"早/晚融合"压成一个数

`retained(d)/R0` 是**累积存活比**(截止第 d 层还没被跨模态读取消费的图像信号比例)。
它的**逐层下降量**才是单层的"融合速率":

**逐层功能融合密度**:
```
f(d) = ( retained(d-1) - retained(d) ) / R0      ( ≥0, Σ f(d) ≈ 1 )
```
= 第 d 层把多少比例的可用图像信号"功能性地"融进答案计算。这是一个深度上的分布。

**Fusion Depth Index(早/晚融合的单一标量)** = f(d) 在相对深度上的质心:
```
FDI = Σ_d (d/N) · f(d)   ∈ [0,1]
```
0 = 完全在输入端(early fusion);1 = 完全在输出端(late fusion)。
**FDI == 报告里的 funcCoM 相对深度。** 判读:early ≲0.33 / mid 0.33–0.66 / late ≳0.66。

实测(VMCBench dev,fig17):**全部 native VLM 都是 mid fusion**——
bee-mix 0.556 / NEO1.0-2B 0.545 / NEO1.0-9B 0.520 / NEO1.5-2B 0.525 /
NEO1.5-9B 0.507 / Gemma-4-12B 0.483。

**关键概念区分**:**架构上的 early/late fusion ≠ 功能上的 FDI**。Native VLM 架构上
是 early fusion(图像 token 从第 0 层就进同一序列),但功能上是 mid fusion
(FDI≈0.5,决定答案的整合在中段)。这是调和"native 架构 = early fusion"与
"功能融合居中段"的关键。

## 8. 注意力质量 ≠ 功能融合(对照 SAIL 2504.10462 / FastV 2403.06764)

SAIL 报告 single-transformer MLLM 给 image token 分配 60–80% 注意力(模块化 10–30%),
测法引用 **FastV(2403.06764)**。我们用同一定义(答案位 → image keys 占比)实测(fig18):
**bee-mix 2.5% / NEO1.0-2B 13%(PT 23%/MT 11%)/ Gemma-4-12B 21%——全在"模块化"区间。**

**关键:FastV 自己的核心发现就是 sink。** FastV 测的正是"图像 token 在深层注意力极低":
浅层(前 2 层)image 拿 ~50%,之后暴跌;**深层 system prompt 吃掉 ~85% 注意力**,
image 只剩零头。这**和我们测到的完全一样**——NEO vision→prefix sink 峰 0.67、
Gemma vision→prefix 峰 0.93、last→prefix 均值 0.34。即**我们的 native 模型表现得就像
FastV 研究的那些模块化模型:有强 sink、image 注意力低**。

所以"为什么和 SAIL 差这么大":
1. **决定 image 占比的是 sink 强度,不是 single-transformer 与否。** 一个模型只要发展出
   强 sink(把预算砸在 system-prompt/anchor 上),image 占比就低。NEO/Gemma 都有强 sink → 落在
   10–30%。SAIL 声称 single-transformer **逃过了** FastV 的 sink(保持 60–80%),但
   **我们三个 native 模型都没逃过** → "single-transformer ⟹ 高 image 注意力"**不是普遍规律**,
   是 SAIL 特定模型的性质,我们是直接反例。
2. **很可能是 system-prompt 口径差异。** FastV 的 sink 专指 **system prompt**;image 占比对
   "输入里有没有 system-prompt/anchor 这个 sink 靶子"极其敏感。若 SAIL 在无/极短 system prompt
   下测,image(占序列多数 token)自然拿到多数注意力 → 60–80%;我们的评测带模板/锚点 token
   (NEO 的 prefix、Gemma 的 boi)→ sink 吸走预算。这是待验证的最大嫌疑。
3. **这不与功能融合结论冲突,反而强化**它:image 注意力只有 10–30%(甚至像 FastV 说的深层趋零),
   这些 native 模型**仍在中段功能融合**(FDI≈0.5、R0≈0.24)。注意力质量(看图多不多)和
   功能融合位置(看图在哪层决定答案)是两个正交的轴——模块化 LLaVA 同样低注意力却也中段融合。

## 9. 融合"窗口":onset(起点)与 completion(终点)—— 回应"FDI 是质心不是起点"的批评

FDI 是 retained 曲线边际下降的**质心**,它把"从第 0 层就开始、拖到尾"和"憋到中间一次性融合"
压成同一个数。要把这两种形状分开,需要**两个方向的封锁**:

- **prefix 封锁(已有)**:切断 [0..d) 层的 text→image 注意力。
  `retained_pre(d)/R0` **下降**的位置 = 融合的**不可恢复点**(最晚必须在哪之前读图)。
  注意:它测的是"已训练电路的依赖位置"——如果模型平时早读图、但被掐断后能在后层补救,
  这条曲线不会在早段下降。**所以它单独不能回答"何时开始融合"。**
- **suffix 封锁(新增,`fusion_window.py` / `*_fusion_window.py`)**:切断 [d..N) 层。
  `retained_suf(d)/R0` 从 0 **上升**到 1;它在深度 d 的值 = **前 d 层已经把多少可用图像信号
  搬进了 text 流**(搬进去的信号之后还能 text→text 传播,所以这是"已完成融合量"的下界紧测度)。
  上升点 = 功能融合的**真实起点**。

两条曲线一起给出**融合窗口** [suffix 上升点, prefix 下降点]:
- 两线几乎重合 → 融合集中在一个窄带内一次完成(无冗余);
- suffix 早升而 prefix 晚降 → 早融合 + 后层有补救能力(冗余);
- 都晚 → 真·晚融合(不是质心伪影)。

索引约定(经独立代码审查逐位验证):`cost[k]` ↔ 封锁 [0..k+1);`suf[k]` ↔ 封锁 [k..N),
`suf[0]` ≡ `cost[N-1]`(全封锁,逐位相同),`suf[N-1]`≈intact。

**row-scope 变体**(`rowscope=lastrow`):只切**答案位**那一行的 text→image(保留 question
token 的读图)。对照全行版本可分解两条通路:image→question tokens(gist 先进问题表征)
vs image→answer 直读。文献(Zhang et al. 2411.18620)在 LLaVA-1.5 上发现 L0-4 有
全图 gist→question-token 转移、且直切 image→last 几乎无损——本变体在 VMCBench 上直接复检。
